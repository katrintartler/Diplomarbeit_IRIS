\section[Wärmesensorik testen]{Wärmesensorik testen\authorkatrin}
Für die Simulation der Temperatur des Kindes, wurden im Testroboter Keramikwiderstände eingebaut, welche als Wärmequellen
genutzt werden. Dazu wurde der Aufbau und die Verschaltung mittels Steckbrett getestet. Besonders zu beachten ist die hohe Leistung 
die für eine Wärmesimulation nötig ist. Dementsprechend ist ein Test auf dem Steckbrett schwierig, da diese meisten nicht für hohe 
Leistung ausgelegt sind. Weiters ist auch die Vesorgung ein Problem, weil die Leistung über eine hohe Stromstärke erbracht werden muss.
\begin{align}
P &= I^2 \cdot R \notag \\
P &= \frac{U^2}{R}
\end{align}
Die verwendeten Keramikwiderstände haben einen Widerstandswert von 2,2$\Omega$. Die Verschaltung erfolgt mit 2 in Serie 
geschalteten Strängen mit jeweils 3 parallel geschalteten Keramikwiderständen. Siehe Abbildung \ref{fig:VerschaltungKRs}, einem Ausschnitt 
des Schaltplans der Platine aus \ref{Testroboter Platinen}. Somit ergibt sich ein Gesamtwiderstand von ca. 1,5$\Omega$. Als Spannungswert werden 7,4V eingesetzt. Diese entsprechen dem Spannungswert eines 2 Zellen 
Lithium-Ionen Akkus. Somit ist ein Nachrüsten auf eine eigenständige Spannungsversorgung möglich siehe \ref{Spannungsversorgung Testroboter}. 
\begin{figure}[htbp]
\centering
\includegraphics[width=10cm]{images/VerschaltungKRs.png}
\caption{Verschaltung der Wärmekomponenten}
\label{fig:VerschaltungKRs}
\end{figure}
\noindent
\begin{align}
P &= \frac{7,4^2}{1,5} = 36,507 W \notag \\ 
I &= \sqrt{\frac{36,507}{1,5}} = 5,0004 A
\end{align}
Der benötigte Strombedarf liegt bei unserer Leistung von 36,507W also bei ca 5A. 
Dieser Bedarf wird von einem 20A Labornetzteil gedeckt. Aufgrund der Wahl dieser Spannungsversorgung wurde beschlossen die weiteren 
Komponenten des Testroboters ebenfalls mit dem Netzteil zu betreiben.

\subsubsection{Interface BLE Kommunikation} %BLE Bleak
Die Kommunikation die per Button Klick gestartet wird is in der Datei ble\_backend.py beschrieben.
Der erste Ansatz war mit der Theorie verbunden, eine normale def callback- Funktion zu schreiben,
welche eine await Funktion mit dem Schreiben des BLE Kommandos beinhalten sollte siehe Listing \ref{lst:ble.py:await}. 
Die await Funktion ist in Python ein Werkzeug um einen Task zu pausieren bis ein anderer beendet ist. 
Das schafft eine Art parallels System \cite{realpython_await}.
\begin{lstlisting}[language=Python, caption={Beispielcode, Falscher Ansatz}, label={lst:ble.py:await}, firstnumber=1]
#callback-funktion 
def heat_on():
    await client.write_gatt_char(CHAR_UUID,b"heat")
\end{lstlisting}
\noindent
Das ist nicht möglich weil ein await nur in einer asynchronen Funktion in Python benutzt werden darf.
Die Callback-Funktion ist aber def.
Daher ist es am Besten mit den Modulen threading und asyncio, als Ergänzung der verwendeten bleak Library 
zu arbeiten, siehe Listing \ref{lst:ble.py:module}.
\begin{lstlisting}[language=Python, caption={Module}, label={lst:ble.py:module}, firstnumber=1]
import threading
import asyncio
from bleak import BleakClient
\end{lstlisting}
\noindent
Zu Beginn müssen die MAC-Adresse des ESP und die Charakter UUID, welche im ESP Code defniert sind, 
auch in der Client Architektur gespeichert werden. Der zweite Schritt ist mithilfe der
Funktionen des Threading-Moduls einen eigenen Thread für BLE zu erstellen. So wird sicher gestellt, dass auch 
beim Senden eines Kommandos aus ble\_backend.py gui.py weiterhin parallel dazu laufen kann. Zu sehen sind diese 
Anforderungen in Listing \ref{lst:ble.py:macusw}.
\begin{lstlisting}[language=Python, caption={MAC, CHAR\_UUID und BLE Thread},label={lst:ble.py:macusw}, firstnumber=5]
ESP_MAC = "a0:b7:65:69:9d:3e" 
CHAR_UUID = "abcd1234-ab12-34cd-56ef-1234567890ab" 
ble_loop = asyncio.new_event_loop()     
\end{lstlisting}
\noindent
asyncio.new\_event\_loop() erstellt uns eine Eventloop mit dem Namen ble\_loop. Diese ist keine der Funktion der threading libraray
sondern hilf uns bei der Anwendung der threading Funktionen für unseren Zweck. 
\noindent
Als nächstes wird mit den Funktionen der bleak library eine Verbindungsfunktion erstellt, siehe Listing \ref{lst:ble.py:connect}.
\begin{lstlisting}[language=Python, caption={Funktion zur Verbindung mit ESP über BLE},label={lst:ble.py:connect},firstnumber=9]
async def ble_connect():
    global client 
    client = BleakClient(ESP_MAC)
    await client.connect()
    print("client_connected") 
\end{lstlisting}
\noindent
Zuerst wird eine Variable client erstellt, diesem client wird mit der Funktion BleakClient(ESP\_MAC), ein Gerät zugewiesen 
(über die MAC-Adresse) mit dem es Verbindung aufnehmen soll. Dementsprechend wird der Laptop im BLE Frequenzbreich nach diesem 
Gerät suchen und dann über das GAP-Protokoll(siehe \ref{GAP}) eine Verbindung aufbauen. Dieser Verbindungsaufbau wird über
await client.connect() initialisiert.Das debugging comment am Ende ist optional. 
\noindent 

\begin{lstlisting}[language=Python, caption={Erstellen und Starten des BLE Threads},label={lst:ble.py:thread},firstnumber=16]
def ble_start_thread():
    def run():
        asyncio.set_event_loop(ble_loop)
        ble_loop.run_until_complete(ble_connect())
        print("BLEloop_running")
        ble_loop.run_forever()
    
    threading.Thread(target=run,daemon=True).start()
\end{lstlisting}
\noindent
Die Funktion ble\_start\_thread (Listing \ref{lst:ble.py:thread}) ist entscheident für die parallel laufenden 
Eigenschaften des Programms. Zuerst wird eine run() Funktion definiert welche dann dem Thread in Zeile 23 als 
Argument übergeben werden kann. Zuerst wird hier dem Term event\_loop unsere zuvor erstellte ble\_loop übergeben. 
Zusätzlich wird sichergestellt das ble\_loop immer läuft. Mit der letzten Zeile der Funktion ble\_start\_thread wird
der Thread zur BLE Kommunikation schließlich gestartet. 
Das nicht blockierende, parallele System ist ab diesem Punkt fertig initialisiert und gestartet. Ab diesem Moment
ist es möglich die eigentliche Funktion (also das Senden der Kommandos) zu implementieren. Die Zeilen davor sind nur eine
Absicherung damit das System fehlerfrei und sauber laufen kann. 
\noindent
Für das Schicken der Kommandos ist die Funktion ble\_send\_cmds verantwortlich. (Listing \ref{lst:ble.py:sending})
\begin{lstlisting}[language=Python, caption={Kommandos über BLE Schnittstelle versenden},label={lst:ble.py:sending},firstnumber=25]
async def ble_send_cmds(data: bytes):   
    if not client.is_connected:
        print("not_connected")
        await client.connect()
    
    await client.write_gatt_char(CHAR_UUID, data)
    print(f"sent{data}cmd") 
\end{lstlisting}
\noindent
Zuerst folgt eine Absicherung auf eine aufrechte Verbindung zwischen ESP und Laptop (Zeilen 26-28).
Für das Versenden eines Kommandos wird nur die Funktion client.write\_gatt\_char(CHAR\_UUID, data)
aus der bleak library benötigt. Hier wird das Kommando das als data Parameter übergeben werden muss, an
die entsprechende Characteristic des Servers (ESPs) geschickt. 

Die verschiedenen Kommdandos die die Callback-Funktionen der Buttons darstellen sind so definiert (Listing \ref{lst:ble.py:callbacks}).
\begin{lstlisting}[language=Python, caption={Beispiel einer Callback-Funktionen der Buttons},label={lst:ble.py:callbacks},firstnumber=35]
def heat_on():
    asyncio.run_coroutine_threadsafe(ble_send_cmds(bytearray([9])), ble_loop)
def heat_off():
    asyncio.run_coroutine_threadsafe(ble_send_cmds(bytearray([10])), ble_loop)
\end{lstlisting}
\noindent 
Zu sehen sind die Callback-Funktionen der Wärmesimulation die in der gui.py als Callbacks für die Heat Buttons aufgerufen werden,
siehe (Listing \ref{lst:gui.py:buttons}). Diese Funktion wird also aufgerufen wenn im Interface jemand einen Heat Button drückt.
Bei dem Button der die Wärmequelle einschalten soll, wird die Zahl 9, als bytearray übermittelt. Nach diesem Muster sind alles Callback-Funktionen 
generiert, sie haben nur entsprechend andere Zahlen die übermittelt werden. Der Grund für die Übertragung der Daten mit bytearrays ist, dass
diese laut ATT Protokoll sowieso in Arrays von Oktets gesendet werden. Wenn man im Code gleiche solche definiert, gibt es 
keine unnötige hin und her Kovertierung, wie es bei zum Beispiel String Kommandos der Fall wäre. \cite{bluetooth_core}\cite{bleak_docu}

\subsection{Server Code} \label{Server Code}
Nachdem die Kommandos gesendet wurden ist es die Aufgabe des ESPs die entsprechenden Kompontenten anzusteuern. 
Für die Programmierung des ESP32 verwenden wir VS-Code mit der Extension PlatformIO. PlatformIO ist eine Entwicklungsumgebung 
welche die wesentlichen Schritte für das Programmieren eines Microcontrollers, wie Kompilieren, Bibliotheksverwaltung und 
Hochladen auf diesen in einer Umgebung zusammenfasst \cite{platformio}. Die Anforderungen um ein Projekt zu erstellen sind die Auswahl eines
gewünschten Frameworks und des Boards welches man verwendet. Der Framework beschreibt die Softwaregrundlage die für die Programmierung 
genutzt werden soll. Damit ist ein großer Teil der Struktur des Projekts konfiguriert \cite{software_framework_wikipedia}. Wird unser ESP als DevBoard eingestellt stehen zwei Frameworkoptionen
zur Auswahl. Der Arduino Framework, welcher auch in der ArduinoIDE genutzt wird,
oder der offizielle Framework von Espressif, ESP-IDF \cite{platformio_frameworks}. Der Framework Arduino stellt die Grundvoraussetzungen bereit, 
mit einer Optimierung auf ESP-IDF hat man zugriff auf eine genauer auf den Microcontroller abgestimmte Funktion, aber es ist ein viel komplexer \cite{espidf_framework}.
In diesem Fall wird der Arduino Framework genutzt \cite{arduino_framework}.

\subsubsection{BLE Verbindung und Kommandoauswertung}
Für die interaktion mit der Bluetoothhardware des ESP wird NimBLE (siehe \ref{NimBLE}) benutzt. Um die Kommunikation richtig 
aufzusetzten wird zuerst eine Art BLE Setup durchgeführt. Wie der Name schon vermuten lässt befindet sich diese Konfiguration im Setup, 
also dem Programmteil welcher immer nur einmal zu anfang ausgeführt wird, bzw. bei jedem neuen flashen oder neustarten \cite{arduino_setup}.
Diese Konfiguration ist in Listing \ref{lst:main.c:blesetup} abgebildet. 
\newpage
\begin{lstlisting}[language=C++,caption={BLE Setup}, label={lst:main.c:blesetup}, firstnumber=126]
  NimBLEDevice::init("Robo"); 

  NimBLEAddress addr = NimBLEDevice::getAddress();
  Serial.print("BLE MAC:");
  Serial.print(addr.toString().c_str()); 
  
  NimBLEServer * server = NimBLEDevice::createServer(); 
  NimBLEService * service = server->createService("12345678-1234-1234-1234-1234567890ab"); 

   NimBLECharacteristic* characteristic =
      service->createCharacteristic(
          "abcd1234-ab12-34cd-56ef-1234567890ab",
          NIMBLE_PROPERTY::READ | NIMBLE_PROPERTY::WRITE
      );

  characteristic->setCallbacks(new CommandCallback());
  characteristic->setValue("ready");

  service->start(); 

  NimBLEAdvertising* adv = NimBLEDevice::getAdvertising();
  adv->addServiceUUID(service->getUUID()); 
  adv->start(); 
\end{lstlisting}
Die Sektion BLE Setup folgt dem allgemeinen Standard zur initialisierung eines BLE-Gerätes. Mit NimBLEDevice::init("Robo"), wird dem Gerät
der von uns gewählte Name Robo zugewiesen. Das Auslesen der MAC-Adresse des ESPs in weiterer Folge ist um die richtige Adresse im ble\_backend.py Code eintragen zu können, wie in Listing 
\ref{lst:ble.py:macusw} angegeben.
Danach wird (Zeile 128 und 129) auf GATT Ebene ein Server erstellt und diesem ein Service zugeschrieben.
Die Erstellung von einem Service ist wichtig für die Funktion der BLE Verbindung (die Funktion eines Services ist in \ref{GATT} nachzulesen), der Service ist mit 12345678-1234-1234-1234-1234567890ab,
der Service\_UUID identifizierbar. Dies ist eine benutzerdefnierte Service UUID die zufällig erstellt wurde. In diesem Service wird eine Charakteristik für das schreiben und lesen 
von Daten zwischen der Laptop-ESP Verbindung erstellt (Zeile 131 -135). Die CHAR UUID die hier angegeben ist muss auch im Client Code definiert sein, siehe \ref{lst:ble.py:macusw}. 
Mit characteristic->setCallbacks(new CommandCallback()) wird festgelegt was passieren soll wenn der Client Nutzdaten schickt. Es wird eine Callback-Funktion defniert die 
ausgeführt werden soll wenn der Server eine Nachricht empfängt. characteristic->setValue("ready") ist eine Nachricht an den Client, sollte dieser die Characteristic lesen, empfängt er "ready".
Mit der Funktion service->start() ist die Konfiguration des Services abgeschlossen, er ist nun auf GATT-Ebene registiert und gestartet. 
In den Zeile 142 bis 144 wir nun die GAP-Ebene angesprochen. Zu Beginn wir der GAP-Zugriff mit  NimBLEAdvertising* adv = NimBLEDevice::getAdvertising() gestartet. Dann der 
Service für den Client dargestellt und dann beginnt der ESP zu werben (adverting), um die Verbindung mit dem Client einzugehen. Verbindungsaufbau in unserem Projekt auch in Abblidung (\ref{fig:ble_gap_bsp})
veranschaulicht. 
\par\vspace{1em}
\noindent
Wie schon oben beschrieben ist CommandCallback() was aufgerufen werden soll, wenn der Client Nutzdaten sendet. 
\begin{lstlisting}[language=C++, caption= {CommandCallback}, label={lst:main.c:commandcb}, firstnumber=53]
class CommandCallback : 
  public NimBLECharacteristicCallbacks {
    void onWrite(NimBLECharacteristic* c, NimBLEConnInfo& coninfo) override {
      
      uint8_t cmd = c->getValue()[0];
      switch(cmd) 
      {
        ...
        case 9:
          heat_on_flag = true;
        break; 

        case 10: 
          heat_off_flag = true; 
        break; 
      }
    }
  }
\end{lstlisting}
\noindent in Listing \ref{lst:main.c:commandcb} zusehen ist die Funktion onWrite(), also wenn die 
Characteristic vom Client beschrieben wird, wird der Wert in die Vairable cmd gespeichert (Zeile 57). 
Je nachdem welcher Button gedrückt wurde wird ein anderer Wert geschickt, wie in Listing \ref{lst:ble.py:callbacks} nachzulesen. 
Dieser wird mithilfe eines switch-case ausgewertet. Jeder Button setzt seine spezifische flag true wenn er gedrückt wird. Empfängt der 
ESP beispielsweise das Kommando für das Einschalten der Wärmequelle wird diese flag gesetzt (Zeile 62). Mithilfe dieser flags erfolgt die 
weitere Reglelung. 

\subsubsection{SchedTask Bibliothek}
Beim Testen der Anwendungen sind öffter Fehler durch das Blockieren von verschiedenen Komponenten durch einzelne 
Tasks aufgetreten. War zum Beispiel ein Motorbewegungs-Task aktiv sind andere Untersysteme blockiert gewesen. 
Zur Verbesserung wurde die Bibliothek SchedTask verwendet. Mit den Funktionen aus SchedTask und in weiterer Folge
aus SchedTaskT können die einzelnen Anwendungen in verschiedene Tasks ausgelagert werden welche dann zeitgesteuert, zyklisch ausgeführt werden. 
So werden die Aufgaben zeitlich und ereignisbasiert ausgeführt.\cite{schedtask_lib}
Dies ist die einfachere Methode im Verglich zur Optimierung auf ein Real-Time System mit FreeRTOS,
mehr dazu in \ref{RTOS}. 
In Listing \ref{lst:main.c:SchedTasks} kann man sehen wie einzelne Tasks erstellt werden. In unserem Fall gibt es
einen Task zur Überprüfung und Auswertung gesendeter Kommandos (dieser beinhaltet auch Audio- und Wärmeansteuerung),
einen Task zur Ansteuerung der Servos und eine der die Temperaturwerte der Wärmequelle überwacht. Dieser ist eine 
Vorsichtsmaßnahme welche nicht zur Gänze fertiggestellt werden konnte. 
\begin{lstlisting}[language=C++, caption= {Loop}, label={lst:main.c:SchedTasks}, firstnumber=184]
SchedTask BLECommands(0,50,BleCmd); 
SchedTask Servos(0,200,ServoSM);
SchedTask Heatcontrol(0, 300, HeatMonitor); 
\end{lstlisting}
\noindent 
Für die Implementierung eines Tasks, wird diesem ein Name zugewiesen, beispielsweise BLECommands,
eine Startzeit, in diesem Fall 0. Somit startet der Task bei 0ms Laufzeit. Der zweite Parameter bestimmt in welchem
Intervall der Task wiederholt werden soll, in diesem Fall alle 50ms. Am Ende wird die Funktion übergeben
die alle 50ms ab 0ms Laufzeit aufgerufen werden soll. Die Funktion BleCmd()
überprüft ob eine der Kommandoflags aus Listing \ref{lst:main.c:commandcb} gestetzt wurde. 
Diese Implementierunf ist in Listing \ref{lst:main.c:SchedTask} zu sehen.
Dadurch haben wir in der klassischen loop() nur eine 
SchedTask dispatcher() Funktion, wie in Listing \ref{lst:main.c:loop} gezeigt. 
\begin{lstlisting}[language=C++, caption= {Loop}, label={lst:main.c:loop}, firstnumber=176]
void loop() {
SchedBase::dispatcher(); 
}
\end{lstlisting}

\subsubsection{Servo Ansteuerung} 
Die Ansteuerung der Servos erfolgt mit der Biblothek ServoEasing \cite{servoeasing_lib}. Die ServoEasing Bibliothek ist speziell für eine sanfte 
Ansteuerung der Motoren ausgelegt. Wenn mittels PWM ein Zielwinkel gesendet wird, ist dieser nicht sofort der Sollwinkel,
sondern wird intern auf mehrer kleine Zielwinkel heruntergerechnet. Der Servo dreht sich also in kleinen Etappen ans Ziel, was einen sanfteren 
Bewegungsablauf hervorruft. Zusätzlich gibt es die Möglichkeit ServoCallback-Funktionen zu implementiern. 
Diese werden über eine Handle Funktion einem Servo zugewiesen. Bei Ankunft des Servos an seiner Zielposition wird diese Callback-Funktion aufgrufen.
Das ist sehr hilfreich wenn man den Servo zum Beispiel für eine gewisse Anzahl von Sequenzen hin und her bewegen will. Der Aufbau dieser
Anwendung mithilfe der ServoEasing Funktionen ist in Listing \ref{lst:main.c:handler} veranschaulicht. Dieses Prinzip wird zusammen  mit einer State Machine für die Ansteuerung der Servos eingesetzt. 
\begin{lstlisting}[language=C++, caption={ServoCallback mit ServoEasing Funktionen}, label={lst:main.c:handler}, firstnumber=160]
// in setup()
Servo1.setTargetPositionReachedHandler(Servo1TargetReachedHandler);
Servo1.setSpeed(90);  

// in State Machine
Servo1TargetReachedHandler(&Servo1);
\end{lstlisting}
\noindent
Jede Callbackfunktion wird einem Servo zugewiesen (Zeile 161), beispielsweise Servo1 (für den Roboter gibt es vier Servos und somit 4 verschiedne ServoCallback-Funktionen), das passiert mithilfe der Funktion 
setTargetPositionReachedHandler() der Übergabeparameter ist die auf die eigene Anwendung programmierte Callback-Funktion. Eine bestimmte Geschwindigkeit 
kann mit Servo1.setSpeed() angegeben werden (Zeile 162). Um den Callback Zyklus zu aktivieren muss die Callback-Funktion einmal aufgerufen werden (beispielhaft in Zeile 165 oder genauer in \ref{lst:main.c:tw} ). 
\par\vspace{1em}
\noindent
Aufgrund der verschiedene Bewegungsmuster die angesteuert werden können, gibt es viele verschiedene Sequenzen wie die Motoren sich bewegen müssen. Deshalb wurde zur implementierung 
eine State Machine aufgebaut siehe Abbildung \ref{fig:statediagramm}.

\noindent
Jeder Zustand beschreibt eine Bewegungsequenz. Wobei es jederzeit möglich ist mehr Bewegungsmuster zu implementieren. Die Bedingung um einen State zu erreichen sind die 
state\_commands, die mit CMD\_x beschrieben sind. 

\paragraph{twitching (TW)} entspricht einer diagonalen Bewegung von einem Arm und einem Bein, abwechselnd mit dem anderen Paar. Es werden also alle vier Servomotoren benötigt.
\paragraph{gross movement (GM)} stellt die Bewegung beider Beine dar. 
\paragraph{localized movement (LM)} hier gibt es einmal die Variante der Bewegung des rechten Beins und einmal die der Bewegung des linken Beins.

}

\noindent 
Die Statemachine selbst ist in der Funktion ServoSM() programmiert, welche als SchedTask aufgebaut alle 200ms 
aufgerufen wird. Gibt es eine Änderung des state\_commands wird der Zustand geändert und der executionstate auf ENTRY gesetzt. 
Dieser Executionstatus Zustand ist aufgrund der zyklischen Wiederholung von ServgoSM() notwending.
Sonst könnte es passieren dass ein State also eine Bewegungssequenz nie ganz durchlaufen kann.
